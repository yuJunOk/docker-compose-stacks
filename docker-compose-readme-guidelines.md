# Docker 子项目 README 生成规范

本文件约束本仓库内各 **Docker Compose / 编排目录** 的文档写法与工程习惯，便于维护者对齐官方资料、避免匿名卷、并让读者在 **Windows / Linux** 上都能按步骤验证。

---

## 1. 与官方文档、GitHub 对齐

- **镜像来源**：README「概览」或「参考」中应给出 **Docker Hub** 或官方镜像文档链接；若项目有 **GitHub 仓库**，给出主仓库或文档站（如 `https://docs.xxx.com/`）。
- **行为说明**：初始化逻辑、环境变量生效时机（例如「仅首次建卷时初始化」）、破坏性操作（删卷、改密码）须 **引用或复述官方说明**，避免与上游行为矛盾。
- **版本**：`docker-compose.yml` 中 `image:` 宜使用 **明确标签**（如 `redis:7-alpine`、`postgres:16-alpine`），避免生产示例长期飘在 `latest`；若必须用 `latest`，在 README 中写明风险。
- **命令与路径**：以官方镜像文档中的容器内路径为准（如 Postgres 数据目录、Redis 配置位置），README 表格与 compose 保持一致。

---

## 2. 卷（持久化）规范

- **禁止依赖匿名卷承载需要保留的数据**。若镜像 Dockerfile 声明了 `VOLUME`（例如 Redis 的 `/data`），必须在 compose 中显式挂载 **命名卷** 或 **绑定挂载**，否则 Docker 会创建 **匿名哈希卷**，难以管理与清理。
- **命名卷**须在顶层 `volumes:` 中声明，并尽量使用 **稳定、可识别的卷名**；需要跨项目区分时可用 `name: xxx` 固定名称，避免随项目目录名变化。
- **纯缓存 / 可丢数据**若故意不落盘，应使用 **`tmpfs`** 或在 README 中明确写「无持久化 / 重启清空」，避免读者误以为有匿名卷即代表「有备份」。
- **绑定挂载**（如 `./data:/data`）须在 README 说明宿主机路径含义、是否在 `.gitignore` 中忽略；数据目录若可通过环境变量配置，应在 `.env.example` 中给出变量名与示例。

---

## 3. README 步骤：简单、顺序符合常理

推荐顺序（与多数读者心理模型一致）：

1. **一句话说明**本目录提供什么、默认部署形态（单机 / 栈）。
2. **概览表**：镜像、端口、数据位置、认证方式、健康检查（如有）。
3. **部署**  
   - **环境变量**：`cp .env.example .env`（或等价说明）→ 必须改的项（密码等）一条写清。  
   - **启动**：一行 **`docker compose up -d`**（或注明工作目录与 `-f`）。  
   - **验证**：见下节；**每个代码块内尽量一行一条命令**，避免把无关命令堆在同一行。
4. **日常使用**：备份、连接串、停止、删卷风险等。
5. **参考链接**：官方 Hub / 文档 / 上游 GitHub。

跨平台复制文件时，可并列写出 **bash** 与 **PowerShell** 一行命令（仅当二者差异大时），不必为每个命令写两套，但验证里 **涉及路径、重定向、curl** 时必须考虑 Windows（见下）。

---

## 4. 验证命令与环境（Windows / Linux / 是否装在 Docker 里）

### 4.1 核心原则

- **验证「容器里的服务是否正常」，优先使用**：

  ```bash
  docker compose ps
  docker compose exec <服务名> <容器内已有的命令>
  ```

  这样 **不要求读者在宿主机安装** `redis-cli`、`psql`、`mysql` 等；与「数据库装在 Docker 里」的常见场景一致。

- **若示例使用宿主机工具**（如本机已装 `redis-cli`、`psql` 连 `127.0.0.1`），必须在 README 中 **明确写出前提**：「宿主机已安装 xxx」或「可选」。避免让读者以为复制命令即可运行。

- **占位符**：文档中可用「将 `REDIS_HOST_PORT` 换成 `.env` 中的值」等说明；**不要**把 `<REDIS_PASSWORD>` 这类占位符当成在终端原样可执行的命令。更稳妥做法是：  
  - 验证节主推 **`docker compose exec redis redis-cli ...`**（密码从容器环境继承或由 `exec -e` 传入）；或  
  - 写明「在 `.env` 同目录执行」并给出 **从文件读取** 的示例（若团队接受），避免在文档里贴真密码。

### 4.2 Windows 与 Linux 差异（写 README 时必须交代）

| 场景 | Windows（PowerShell / CMD） | Linux / macOS |
|------|-----------------------------|----------------|
| 丢弃 curl 输出只看状态码 | `curl.exe -s -o NUL -w "%{http_code}\n" URL` | `curl -s -o /dev/null -w "%{http_code}\n" URL` |
| PowerShell 中 `curl` 常为别名 | 使用 **`curl.exe`** 调用真实 curl | 通常直接使用 `curl` |
| 路径 | 绑定挂载可用 `D:/data/xxx` 等形式便于 Docker 解析 | 同左或 POSIX 路径 |

验证小节建议采用 **「主路径（exec）+ 附注（宿主机工具可选）」** 双轨表述，与 **`pgsql-standalone`** README 中「容器内 `psql` / 宿主机若有 `psql`」的写法对齐。

### 4.3 健康状态

- 若 compose 配置了 **`healthcheck`**，验证步骤中应说明：**`docker compose ps` 中状态为 `healthy` 即通过**（并注明首次就绪可能需要数秒）。

## 5. README 推荐章节框架（复制模板）

```markdown
# <服务名>（Docker Compose）

一句话说明 + 官方/文档链接。

## 概览

| 项目 | 说明 |
|------|------|
| 镜像 | … |
| 端口 | … |
| 数据 / 卷 | 命名卷或绑定挂载路径（勿匿名持久化） |
| 认证 | … |
| 健康检查 | … |
| 可视化 / 管理界面 | 有则写 URL 与端口；无则写「无官方 Web，可用 xxx 客户端连本机端口」 |

## 部署

### 1. 环境变量

一行 `cp` / `Copy-Item`（如需要两套则各一行）。

### 2. 启动

docker compose up -d

### 3. 验证

- docker compose ps（+ healthy 说明）
- docker compose exec …（不依赖宿主机 CLI 的主路径）
- （可选）宿主机工具 / 浏览器 / curl（注明 Windows 用 NUL、curl.exe）

## 日常使用

备份、停止、删卷警告、安全提示（绑定 0.0.0.0 等）。

## 参考链接

- Docker Hub / 官方文档 / GitHub
```

## 6. 修订记录（可选）

重大变更（例如统一卷命名、验证命令改为全 `exec`）可在子项目 README 底部用一行日期说明，或在 Git 提交信息中写清即可。

本规范自身若与某个子项目现状不一致，**以本规范为准逐步收敛子项目**；冲突时可在子项目 README「注意」中短期注明例外与原因。
